/opt/maat/bin
----------------------------------------------

Maat top-level executables. The AM executable, attestmgr, is the only interesting binary here.
It is given the `attestmgr_exe_t` to allow it to enter the special `attestmgr_t` domain.
In common operation, attestmgr should be run as root to allow binding to a privileged port.
After initialization, attestmgr will setuid/setgid to the Maat user (as specified in its
configuration file or on the command line).

/opt/maat/etc/maat/credentials
----------------------------------------------

Maat credentials directory.
This directory contains certificates and private keys used by Maat. Its contents should have type
`maat_certificate_t` or `maat_private_key_t` respectively. Public key certificates should be owned by
root:root (or root:wheel) but may be readable by anyone (chmod 0644). Private key files should be
readable only by the Maat user, it is suggested that they are owned by root:maat and set to user/group
read only (chmod 0440).

+------------+------+-------+-----------------------------------------------------+----------------------------------------+
|UNIX Perms  |User  |Group  |SELinux Label                                        |Filename                                |
+============+======+=======+=====================================================+========================================+
|-rw-r--r--. |root  |root   |system_u:object_r:maat_certificate_t:s0              |ca.pem                                  |
+------------+------+-------+-----------------------------------------------------+----------------------------------------+

/opt/maat/lib
----------------------------------------------

System library directory.
This is the default system library directory for the Maat prefix. The Maat runtime libraries are
installed directly in this directory, and subdirectories are created for ASP & APB implementations.
Currently these libraries are all installed with the default `lib_t` type. This is reasonable
because allowing other applications to map the Maat libraries is not problematic. However, it
does mean that Maat's various domain types may map any library with this type, which may be
undesirable (e.g., because a malicious local administrator may modify the library search path to cause
Maat to map backdoored versions of these libraries). In the future we may move to a specialized
library type to avoid this, but it would require distributing Maat with all of its own dependencies
(including its own runtime linker/loader stub which seems like a huge pain).

/opt/maat/lib/maat/apbs
----------------------------------------------

APB executable installation directory.
Most APB executables are given a unique SELinux type, the SELinux policy module maat.pp defines
type transition rules to allow these types to enter APB specific domains at runtime.
These domains should be granted access to the exact set of Measurement Specification types
supported by the APB, and to the ASPs used by the APB to fulfill them.

APB executables may be installed with the `default_apb_exe_t` type, in which case they will be run
in the `default_apb_t` domain and given access to only the default set of measurement specifications
and ASPs. Because the default ASPs are not granted any interesting permissions, this is rarely
desirable.

This directory itself should have type `apb_dir_t`

This directory and its contents must be readable/executable (but not writable) by the Maat user to allow
the AM to execute any APB. Because well-behaved APBs do not perform any privileged operations,
unlike ASPS, none of these executables is expected to have its SUID or SGID bits set.

/opt/maat/lib/maat/asps
----------------------------------------------

ASP executable installation directory.
Most ASP executables are given a unique SELinux type. The SELinux policy module maat.pp defines
type transition rules to allow these types to enter the ASP specific domains at runtime.
The allow_apb_asp(abp_t, asp_exe_t, asp_t) macro defined in libmaat-asp.pp allows APBs in the domain
`apb_t` to execute ASPs installed with file type `asp_exe_t` in the domain `asp_t`.

ASPs that do not require any non-default privileges (see above) may be installed with the
`default_asp_exe_t` and run in the `default_asp_t` domain, and they will be executable by any APB.

This directory and its contents must be readable/executable (but not writable) by the Maat user to allow
the AM and APBs to execute the ASPs. ASPs requiring special permissions (e.g., reading process
state) should be installed with appropriate ownership, and their SUID or SGID bits set.
